# 서비스 통합 테스트 체크리스트 (Frontend-Backend)

백엔드 기능 구현 후, 실제 서비스 동작을 확인하기 위한 점검 항목입니다.
---

## 스타일 (Style) 관련 기능
사용자가 스타일을 등록하고, 관리하고, 조회하는 핵심 로직을 점검합니다.

### 스타일 등록 및 수정/삭제
- [] **다중 이미지 업로드:** 여러 장의 사진이 순서(`order`)에 맞게 업로드되는가?
- [] **태그 제한:** 태그가 최대 3개까지만 등록되는가? (4개 이상 시 차단 여부)
- [] **스타일 구성(Items):** 상의, 하의, 아우터, 원피스, 신발, 가방, 패션잡화 각각의 의상명/브랜드/가격이 정확히 저장되는가?
- [] **비밀번호 보호 (수정):** 등록 시 비밀번호와 일치할 때만 수정 폼이 활성화되는가?
- [] **비밀번호 보호 (삭제):** 비밀번호가 틀릴 경우 `403 Forbidden` 에러와 함께 삭제가 거부되는가?

### 1-2. 스타일 목록 및 조회
- [] **데이터 바인딩:** 대표 이미지(첫 번째 사진), 제목, 닉네임, 태그, 구성, 조회수, 큐레이팅 수가 누락 없이 표시되는가?
- [] **정렬 로직:** 최신순, 조회순, 큐레이팅순 클릭 시 목록이 정확히 재배열되는가?
- [] **검색 기능:** 닉네임, 제목, 상세설명, 태그 키워드로 검색 시 정확한 필터링 결과가 나오는가?
- [] **페이지네이션:** 스크롤 또는 페이지 이동 시 `cursor` 기반으로 데이터가 중복 없이 로드되는가?

---

## 큐레이팅 관련 기능
스타일에 대한 유저들의 평가와 점수 합산 로직을 점검합니다.

- [] **평가 등록:** 4대 지표(트렌디, 개성, 실용성, 가성비) 점수와 한줄평이 정상 저장되는가?
- [] **상세 페이지 연동:** 스타일 상세 조회 시 해당 스타일에 달린 큐레이팅 목록이 모두 노출되는가?
- [] **검색:** 큐레이팅 내 닉네임 및 내용 검색이 작동하는가?
- [] **수정/삭제 권한:** 큐레이팅 등록 시 설정한 비밀번호로만 수정/삭제가 가능한가?
- [] **답글 포함 조회:** 큐레이팅 데이터 호출 시 하위 답글(Reply) 정보가 포함되어 오는가?

---

## 답글 관련 기능
스타일 작성자가 큐레이팅에 남기는 피드백 기능을 점검합니다.

- [] **작성 권한 확인:** **스타일 등록 시의 비밀번호**와 일치해야만 답글 등록이 가능한가? (주인 인증)
- [] **1개 제한 로직:** 이미 답글이 등록된 큐레이팅에 추가 답글 작성이 금지되는가?
- [] **수정/삭제:** 답글 등록 시 설정한 비밀번호로 본인 확인 후 수정/삭제가 가능한가?
- [] **데이터 바인딩:** 큐레이팅 항목 바로 아래에 작성자 닉네임과 답글 내용이 정확히 표시되는가?

---

## 랭킹 관련 기능
백엔드 점수 계산 로직이 프론트엔드 UI에 정확히 반영되는지 점검합니다.

- [] **기준별 정렬:** 전체, 트렌디, 개성, 실용성, 가성비 탭 클릭 시 해당 점수 기준 내림차순으로 정렬되는가?
- [] **순위 표시:** 계산된 랭킹(`ranking`)과 평균 평점(`rating`)이 UI에 올바르게 노출되는가?
- [] **페이지네이션:** 랭킹 목록에서도 커서 기반 페이징이 정상 작동하는가?
- [] **필터링:** 큐레이팅이 없는 스타일은 랭킹 목록에서 자동으로 제외되는가?

---

## 이미지
- [] **대표 이미지 설정:** 목록 화면에서 항상 업로드된 사진 중 첫 번째 사진이 썸네일로 나오는가?
- [] **인기 태그 클릭:** 갤러리 상단 인기 태그 클릭 시 해당 태그가 포함된 스타일 목록으로 즉시 필터링되는가?
- [] **상세 페이지 슬라이드:** 상세 조회 시 여러 장의 이미지를 슬라이드나 리스트로 모두 확인할 수 있는가?

## 태그
- [] **태그 조회:** 스타일에 등록된 태그가 보이는가?
- [] **태그 등록:** 태그 등록시 최대 3개까지 제한하고 있는가?
---

## 공통 점검 사항
- [] **BigInt 처리:** 모든 ID 값이 프론트엔드에서 깨지지 않도록 문자열(String)로 전달되는가?
- [] **에러 핸들링:** 비밀번호 오류(403), 데이터 없음(404) 등의 상황에서 프론트에 적절한 에러 객체가 전달되는가?
